推荐链路

DoclingParser 解析原文档，拿到 DoclingDocument
导出 markdown = doc.export_to_markdown()
用 MarkdownHeaderTextSplitter 先按标题层级切大块
对每个大块再做一次小块切分
为每个 chunk 生成 embedding
写入 Qdrant，payload 里保存文本和元数据
检索时先向量召回，再按 metadata 做重排或过滤
为什么这么做

Docling -> Markdown 能最大限度保留标题、列表、表格等轻结构
MarkdownHeaderTextSplitter 适合先按语义章节切块，比纯字符切分稳定
只靠 header splitter 不够，因为某些章节会很长，还需要第二层小块切分
向量库里必须保留足够 metadata，否则后面很难做可追溯检索和结果展示
我会怎么分层

parser.py
输入文件路径或 URL
输出统一文档对象：source、markdown、text、metadata
chunker.py
第一阶段：MarkdownHeaderTextSplitter
第二阶段：对超长块再做递归切分
输出 chunk 列表，每个 chunk 带 header 路径和位置信息
embedder.py
只负责把 list[str] -> list[list[float]]
这里和 LiteLLM / DashScope 解耦
indexer.py
负责 chunk + vector -> Qdrant points
retriever.py
负责 query embedding + Qdrant search
我建议保留的 metadata
每个 chunk 至少保留：

source
doc_id
chunk_id
text
headers
section_title
chunk_index
start_offset
end_offset
content_type
created_at
如果 Docling 能拿到更细信息，还可以加：

page_numbers
tables
images
language
chunk 策略
我不会只用 MarkdownHeaderTextSplitter 一步到位，因为它解决的是“按结构切”，不是“按模型上下文长度切”。

更稳的做法：

先按标题切出章节块
再对章节块按长度二次切分
chunk size 先从 500 到 1000 tokens 对应文本长度试起
overlap 保留 10% 到 20%
embedding 输入用什么
我会优先用“清洗后的 chunk text”，不是整段原始 Markdown。

通常做法：

保留标题文本
去掉无意义空白
表格转成尽量可读文本
让 chunk 内容保持自然语言可检索
也就是说：

存储 raw_markdown
embedding 用 embedding_text
展示给用户时可返回 raw_markdown 或原始片段
Qdrant 入库结构
我会把每个 chunk 存成一个 point：

id: 稳定 chunk id
vector: embedding 向量
payload:
source
doc_id
chunk_id
text
headers
section_title
chunk_index
这样检索出来后可以直接展示，也方便按文档聚合。

我会避免的坑

不直接把整个 PDF 导成一大段文本再 embedding
不只存向量，不存原文和 metadata
不只按字符硬切，不看 Markdown 结构
不把 Markdown 原样全量喂 embedding，表格和格式噪音会影响效果
不让 chunk id 每次重建都变化，否则增量更新会很麻烦
最小可行实现
先做这条最短链路：

DoclingDocument
export_to_markdown()
MarkdownHeaderTextSplitter
二次小块切分
embedding
Qdrant upsert
先不做：

OCR 参数路由
表格专门索引
图片理解
reranker
hybrid search
