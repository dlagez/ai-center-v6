**文档名称**：清欠表 AI 填报实现逻辑  
**编写日期**：2026-04-13  
**适用对象**：产品、后端、前端、测试

---

# 1. 当前实现目标

当前版本已经实现一条可用的 Excel 更新链路，目标是：

1. 上传一份目标 Excel，创建一个持久任务。
2. 在同一个任务下反复执行多次更新操作。
3. 每次操作都基于上一次生成的 Excel 继续更新，不是临时结果。
4. 每次操作都保留独立日志和独立输出文件，可单独下载。
5. 前端按“任务列表页 + 任务详情页”组织。

当前实现重点已经从“单次执行”升级为“任务制、多次操作、持续更新”。

---

# 2. 当前整体流程

当前真实流程如下：

```text
用户打开任务列表页
-> 上传 Excel
-> 调用 POST /workflow/excel-update/tasks
-> 服务端创建任务目录并保存 source Excel
-> 返回 task_id
-> 用户进入任务详情页
-> 输入自然语言操作指令
-> 调用 POST /workflow/excel-update/tasks/{task_id}/operations
-> 服务端根据当前任务最近一次 Excel 继续执行
-> 如缺少关键参数，则调用 analyzer 自动识别
-> workflow 固定进入取数 -> 更新 Excel -> 导出新文件
-> 保存本次 operation 日志和输出文件
-> 前端展示日志、识别结果、下载入口
```

# 3. 页面结构

## 3.1 任务列表页

页面入口：

* `/pages/excel-update/tasks`

页面职责：

1. 创建新任务。
2. 展示任务表格。
3. 点击整行进入任务详情页。

当前列表页特征：

1. 新建任务和任务表格为上下布局。
2. 表格每一行都可点击进入详情。
3. 表格展示字段包括：
   * 文件名
   * 任务 ID
   * 操作次数
   * 最新列
   * 更新时间

## 3.2 任务详情页

页面入口：

* `/pages/excel-update?task_id={task_id}`

页面职责：

1. 展示任务类型卡片。
2. 对当前任务继续执行操作。
3. 展示操作日志。
4. 展示任务级汇总信息。

当前详情页结构：

1. 顶部是任务类型卡片。
   当前只有一个任务类别：`Excel 批量更新`
2. 中间是操作区。
3. 下面是日志形式的操作列表。
4. 最下面是任务详情摘要。

# 4. 当前前端交互逻辑

## 4.1 创建任务

用户在任务列表页上传一个 Excel 后：

1. 前端调用 `POST /workflow/excel-update/tasks`
2. 服务端创建一个新的任务
3. 跳转到任务详情页

注意：

1. 当前创建任务时不执行更新。
2. 创建任务只是保存目标 Excel，并初始化任务元数据。

## 4.2 执行操作

用户在任务详情页只需要输入：

1. 自然语言提示词

前端当前不再展示以下字段：

1. Excel 匹配列
2. 业务匹配字段
3. 查询条件 JSON
4. 操作人
5. 允许覆盖已有值

其中：

1. `match_column` 当前固定走后端默认值 `项目编号`
2. `match_field` 当前固定走后端默认值 `project_no`
3. 其他隐藏字段当前也走后端默认逻辑

## 4.3 识别结果展示

当前详情页中：

1. 不再单独保留“先识别参数”按钮。
2. 执行操作后，如果本次有自动识别，页面右下会弹出日志提示。
3. 弹出内容主要展示：
   * 识别到的 `sheet页`
   * 识别到的 `目标列名`
4. 每条操作日志中也会保留这次识别结果。

# 5. 当前 API 设计

## 5.1 创建任务

接口：

* `POST /workflow/excel-update/tasks`

职责：

1. 接收上传的目标 Excel。
2. 保存到 `_uploads` 临时目录。
3. 在任务目录中复制并保存 source Excel。
4. 生成 `task_id`。
5. 返回任务详情。

## 5.2 查询任务列表

接口：

* `GET /workflow/excel-update/tasks`

职责：

1. 遍历任务目录。
2. 读取每个任务的 `result.json`。
3. 返回任务摘要列表。

注意：

1. 当前列表接口按新结构严格读取，不再兼容旧版 `result.json`。

## 5.3 查询任务详情

接口：

* `GET /workflow/excel-update/tasks/{task_id}`

职责：

1. 返回任务完整详情。
2. 包括：
   * 任务基本信息
   * 当前 Excel 路径
   * 操作次数
   * 全部 operation 日志

## 5.4 新增一次操作

接口：

* `POST /workflow/excel-update/tasks/{task_id}/operations`

职责：

1. 接收当前这次自然语言操作指令。
2. 如果缺少关键参数，则自动分析：
   * `sheet_name`
   * `target_column`
   * `query_conditions`
3. 基于当前任务最近一次输出文件继续执行更新。
4. 保存本次操作日志和输出文件。

## 5.5 下载最新文件

接口：

* `GET /workflow/excel-update/tasks/{task_id}/file`

职责：

1. 下载任务当前最新 Excel。

## 5.6 下载某次操作文件

接口：

* `GET /workflow/excel-update/tasks/{task_id}/operations/{operation_id}/file`

职责：

1. 下载某一次 operation 对应生成的 Excel 文件。

# 6. 当前后端模块职责

当前核心实现位于：

* `src/workflow/excel_update/`

## 6.1 `task_service.py`

职责：

1. 创建任务目录。
2. 保存 source Excel。
3. 维护任务元数据。
4. 在一个任务下追加多次 operation。
5. 维护当前最新 Excel 路径。
6. 提供任务级和操作级下载路径。

当前任务目录结构的核心特点：

1. 每个任务一个目录。
2. 目录内保存：
   * source Excel
   * 每次操作输出文件
   * `result.json`

## 6.2 `service.py`

职责：

1. 解析 Excel。
2. 取业务数据。
3. 更新 Excel。
4. 导出新的 Excel。
5. 汇总执行结果。

当前 `service.py` 里的流程是固定的：

1. `parser`
2. `fetcher`
3. `updater`
4. `exporter`

也就是说，只要执行一次 operation，就会固定进入取数流程。

## 6.3 `analyzer.py`

职责：

1. 理解自然语言提示词。
2. 识别目标 `sheet页`
3. 识别目标列名
4. 推断查询条件

当前 analyzer 的定位：

1. 不是独立操作步骤。
2. 而是在执行 operation 时按需触发。

触发条件：

1. 用户传了 `user_prompt`
2. 且关键字段没有完整提供

## 6.4 `fetcher.py`

职责：

1. 根据 `query_conditions` 获取业务数据。

当前状态：

1. 当前仍是占位实现。
2. 不是调用真实 PM API。
3. 默认返回两条模拟数据：

```json
[
  { "project_no": "HKZC-N-YW-2021-001", "value": 20 },
  { "project_no": "HKZC-N-YW-2021-002", "value": 30 }
]
```

因此，当前真实逻辑是：

1. 执行时固定调用 `fetch_business_records`
2. 但这个调用目前是模拟取数，不是真实接口

## 6.5 `parser.py`

职责：

1. 打开 Excel
2. 定位目标 sheet
3. 定位匹配列与目标列
4. 建立 `项目编号 -> 行号` 映射

当前特性：

1. 支持多级表头。
2. 支持跨行识别列头。
3. 自动跳过表头下空行。

## 6.6 `updater.py`

职责：

1. 遍历业务 records
2. 按 `project_no -> 项目编号` 匹配 Excel 行
3. 将 `value` 写入目标列
4. 记录变更、错误和未匹配项

## 6.7 `exporter.py`

职责：

1. 将更新后的 workbook 导出成新的 Excel 文件。

# 7. 当前任务与操作数据模型

## 7.1 任务

当前系统已将“任务”和“操作”拆开。

任务层包含：

1. `task_id`
2. 原始文件名
3. source Excel 路径
4. current Excel 路径
5. 最新输出文件名
6. 操作次数
7. 操作列表

## 7.2 操作

每次 operation 包含：

1. `operation_id`
2. 操作序号
3. 创建时间
4. 本次请求参数
5. 本次自动识别结果
6. 本次执行结果
7. 本次输出文件名
8. 本次下载地址

---

# 8. 当前日志展示逻辑

详情页中的操作日志已经实现为 log 形式。

每条日志当前展示：

1. 第几次操作
2. 执行时间
3. 识别到的 `sheet页`
4. 识别到的 `目标列名`
5. 更新单元格数量
6. 匹配记录数
7. 错误数
8. 操作参数
9. 变更记录
10. 错误和未匹配项
11. 本次表格下载入口

当前日志能力已经满足：

1. 每次操作独立留痕
2. 每次操作产物独立下载
3. 可追溯识别结果和执行结果

---

# 9. 当前限制

当前实现仍有以下限制：

1. 执行 operation 时固定进入取数流程。
2. `fetcher.py` 目前还是模拟数据，不是真实 PM 接口。
3. 当前只支持一种任务类型：`Excel 批量更新`
4. 当前尚未支持通过另一个 Excel 作为数据源。
5. 当前尚未实现“大模型判断是否需要调用 PM API”的路由层。

---

# 10. 当前结论

当前版本已经实现的不是“单次识别 + 单次执行”的旧流程，而是：

1. 任务制
2. 多次操作
3. 持久化结果
4. 日志式详情页
5. 每次操作独立表格产物

但取数逻辑目前仍然是固定链路：

1. 执行 operation
2. 调用 `fetch_business_records`
3. 写入 Excel

后续如果要支持：

1. 大模型决定是否调用 PM
2. 通过 Excel 作为数据源
3. 多种任务类型卡片

则需要在当前实现上继续增加“路由层”和“数据源层”。
