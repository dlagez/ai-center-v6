# 清欠表 AI 填报实现逻辑

**文档名称**：清欠表 AI 填报实现逻辑  
**编写日期**：2026-04-10  
**适用对象**：产品、后端、前端、测试

---

# 1. 实现目标

当前实现目标是提供一条可联调的完整链路：

1. 用户上传 Excel 文件；
2. 用户通过自然语言表达更新诉求，例如“把这张清欠表 3 月实际产值更新一下”；
3. 大模型理解用户意图，并识别需要查询的业务数据、目标工作表、匹配列和目标列；
4. 系统根据大模型确认后的结构化参数获取业务数据；
5. 系统按“Excel 中的项目编号列”与“业务数据中的项目编号字段”进行匹配；
6. 将业务数据中的数值批量写入 Excel 指定目标列；
7. 返回任务结果摘要，并提供更新后 Excel 下载入口。

当前实现采用**大模型负责理解 + 规则流程负责执行**的模式，不使用大模型直接修改 Excel。

---

# 2. 整体流程

整体流程如下：

```text
用户打开页面
→ 上传 Excel 文件并输入自然语言指令
→ 大模型解析任务意图
→ 大模型识别查询条件
→ 调用业务系统 API 获取数据
→ 大模型读取所有 sheet 并识别目标 sheet
→ 大模型读取表头并识别匹配列、目标列
→ 用户确认识别结果
→ 提交 /workflow/excel-update/tasks
→ 服务端保存上传文件
→ 创建更新任务
→ 调用 fetcher 获取业务数据
→ 解析 Excel 表头和项目编号列
→ 按项目编号批量匹配行
→ 将业务数据 value 写入 target_column
→ 保存新的 Excel 文件
→ 返回 task_id、摘要、下载地址
→ 用户下载更新后的 Excel
```

---

# 3. 页面层逻辑

页面入口：

* `/pages/excel-update`

页面提交的核心参数如下：

* `file`：上传的 Excel 文件
* `user_prompt`：用户自然语言指令，例如“把这张清欠表 3 月实际产值更新一下”
* `sheet_name`：目标工作表名称，可为空
* `match_column`：Excel 中用于匹配的列名，当前默认值为 `项目编号`
* `match_field`：业务数据中用于匹配的字段名，当前默认值为 `project_no`
* `target_column`：Excel 中需要写入的目标列名，例如 `3月实际产值`
* `query_conditions`：查询业务数据的条件，JSON 数组格式
* `overwrite_existing`：是否允许覆盖 Excel 原值
* `operator`：操作人，可选

页面职责分两部分：

1. 采集用户自然语言指令和文件；
2. 展示大模型识别结果，并在用户确认后触发更新任务。

---

# 4. API 层逻辑

当前对外提供三组接口：

## 4.1 创建任务

接口：

* `POST /workflow/excel-update/tasks`

职责：

1. 接收上传文件和表单参数；
2. 调用大模型解析用户意图；
3. 输出结构化更新参数；
4. 将上传文件保存到服务端目录；
5. 组装内部 `ExcelUpdateRequest`；
6. 调用任务服务执行更新；
7. 返回任务结果、摘要和下载地址。

## 4.2 查询任务详情

接口：

* `GET /workflow/excel-update/tasks/{task_id}`

职责：

* 返回任务执行结果，包括汇总、未匹配项目、错误信息、变更记录等。

## 4.3 下载结果文件

接口：

* `GET /workflow/excel-update/tasks/{task_id}/file`

职责：

* 下载更新后的 Excel 文件。

# 5. 工作流模块划分

当前实现位于：

* `src/workflow/excel_update/`

模块职责如下：

## 5.1 `task_service.py`

负责任务级管理：

* 生成 `task_id`
* 创建任务目录
* 调用工作流服务
* 保存 `result.json`
* 提供结果文件查询和下载路径

## 5.2 `service.py`

负责串联完整执行流程：

1. 接收大模型确认后的结构化参数；
2. 解析 Excel；
3. 获取业务数据；
4. 执行批量更新；
5. 导出更新后 Excel；
6. 汇总结果。

## 5.3 大模型解析层

大模型在当前方案中的职责不是直接修改 Excel，而是负责“理解”。

主要负责四件事：

1. 理解用户自然语言指令；
2. 生成业务系统 API 查询条件；
3. 在多个 sheet 中识别目标 sheet；
4. 在表头中识别 `match_column` 和 `target_column`。

例如用户输入：

* `把这张清欠表 3 月实际产值更新一下`

大模型需要识别出如下信息：

```json
{
  "query_conditions": [
    { "key": "month", "value": "2026-03" },
    { "key": "metric", "value": "actual_output" }
  ],
  "sheet_name": "软件项目清欠表-收付现",
  "match_column": "项目编号",
  "match_field": "project_no",
  "target_column": "3月实际产值"
}
```

## 5.4 `fetcher.py`

负责查询业务系统数据。

当前是模拟实现，固定返回两条数据：

```text
HKZC-N-YW-2021-001 -> 20
HKZC-N-YW-2021-002 -> 30
```

对应返回结构为：

```json
[
  { "project_no": "HKZC-N-YW-2021-001", "value": 20 },
  { "project_no": "HKZC-N-YW-2021-002", "value": 30 }
]
```

后续接入真实业务系统时，只需要替换该模块。

## 5.5 `parser.py`

负责解析 Excel 模板。

当前逻辑：

1. 使用 `openpyxl` 打开 Excel；
2. 按 `sheet_name` 定位工作表，未传则使用默认激活页；
3. 在前 10 行中扫描表头；
4. 匹配时忽略空格和换行；
5. `match_column` 和 `target_column` 不要求位于同一行；
6. 支持多级表头场景，例如 `项目编号` 在第 2 行、`3月\n实际产值` 在第 3 行；
5. 记录：
   * 表头所在行
   * 项目编号列位置
   * 目标列位置
7. 自动跳过表头下方空行，找到数据起始行；
8. 建立 `项目编号 -> Excel 行号` 映射；
9. 记录 Excel 中重复项目编号。

## 5.6 `updater.py`

负责执行批量写入。

当前逻辑：

1. 遍历 `fetcher.py` 返回的业务数据；
2. 用 `match_field` 从业务记录中取项目编号；
3. 到 Excel 的 `项目编号 -> 行号` 映射中查找对应行；
4. 找到后取业务记录中的 `value`；
5. 将 `value` 写入 `target_column` 对应列；
6. 记录旧值、新值、行号和列名；
7. 输出：
   * `changes`
   * `errors`
   * `unmatched_keys`

## 5.7 `exporter.py`

负责保存更新后的工作簿。

当前逻辑：

* 将内存中的 workbook 保存为新的输出文件，不覆盖原上传文件。

# 6. 大模型确认逻辑

当前方案中，大模型负责“确认更新参数”，规则流程负责“执行更新”。

## 6.1 用户输入

用户输入示例：

* `把这张清欠表 3 月实际产值更新一下`

## 6.2 第一步：识别需要查询的数据

大模型根据用户输入，判断当前要更新的是：

* 时间：`2026-03`
* 指标：`actual_output`

并生成查询条件：

```json
[
  { "key": "month", "value": "2026-03" },
  { "key": "metric", "value": "actual_output" }
]
```

系统据此调用业务系统 API，拿到类似数据：

```json
[
  { "project_no": "HKZC-N-YW-2021-001", "value": 20 },
  { "project_no": "HKZC-N-YW-2021-002", "value": 30 }
]
```

## 6.3 第二步：识别目标 sheet

大模型读取 Excel 的全部 sheet 名，找到最符合用户意图的目标工作表。

例如识别出：

* `软件项目清欠表-收付现`

## 6.4 第三步：识别表头

大模型读取目标 sheet 的表头区域，识别：

* 哪一列是 `项目编号`
* 哪一列是 `3月实际产值`

并输出：

```json
{
  "sheet_name": "软件项目清欠表-收付现",
  "match_column": "项目编号",
  "match_field": "project_no",
  "target_column": "3月实际产值"
}
```

## 6.5 第四步：用户确认

页面向用户展示大模型识别结果，用户确认后再执行更新。

确认信息至少包括：

* 目标 sheet
* Excel 匹配列
* 业务匹配字段
* 目标列
* 查询条件

## 6.6 第五步：规则流程执行更新

确认完成后，现有更新引擎执行：

1. 调用业务系统 API 获取数据；
2. 按 `match_field` 读取业务记录中的项目编号；
3. 按 `match_column` 定位 Excel 中的项目编号列；
4. 匹配项目所在行；
5. 将 `value` 写入 `target_column`。

---

# 7. 匹配逻辑说明

当前匹配逻辑已拆分为两个概念：

## 6.1 `match_column`

含义：

* Excel 中的匹配列名

示例：

* `项目编号`

用途：

* 用于在 Excel 中定位“哪个列存放项目编号”

## 6.2 `match_field`

含义：

* 业务数据中的匹配字段名

示例：

* `project_no`

用途：

* 用于从业务数据中读取项目编号值

## 6.3 当前匹配过程

示例 Excel：

| 项目编号 | 项目名称 | 3月实际产值 |
|---|---|---|
| HKZC-N-YW-2021-001 | 项目A |  |
| HKZC-N-YW-2021-002 | 项目B |  |

示例业务数据：

```json
[
  { "project_no": "HKZC-N-YW-2021-001", "value": 20 },
  { "project_no": "HKZC-N-YW-2021-002", "value": 30 }
]
```

当参数为：

* `match_column = 项目编号`
* `match_field = project_no`
* `target_column = 3月实际产值`

系统会执行：

1. 在 Excel 中找到“项目编号”列；
2. 在业务数据中读取 `project_no`；
3. 用两边的项目编号值做匹配；
4. 将 `value` 写入“3月实际产值”列。

---

# 8. 写入逻辑说明

当前写入规则如下：

1. 一行代表一个项目；
2. 项目编号是一整列；
3. 目标列通过 `target_column` 指定；
4. 每条业务数据只写入一个值字段，即 `value`；
5. 如果 Excel 中找不到对应项目编号，则记入 `unmatched_keys`；
6. 如果业务数据中缺少匹配字段，则记入 `errors`；
7. 如果业务数据中缺少 `value`，则记入 `errors`；
8. 如果 Excel 中项目编号重复，则记入 `errors`；
9. 如果 `overwrite_existing=false` 且目标单元格已有值，则跳过并记入 `errors`；
10. 如果旧值与新值相同，则不重复写入。

# 9. 输出结果说明

当前任务执行完成后，会输出：

## 8.1 更新后 Excel 文件

保存位置：

* `data/excel_update/<task_id>/`

文件名示例：

* `原文件名_updated.xlsx`

## 8.2 任务结果 JSON

保存位置：

* `data/excel_update/<task_id>/result.json`

包含内容：

* `task_id`
* 原文件名
* 输出文件名
* 下载地址
* 更新结果摘要
* 未匹配项目
* 错误信息
* 变更记录

## 8.3 返回给前端的摘要

当前汇总字段包括：

* `total_records`
* `matched_records`
* `updated_cells`
* `skipped_records`
* `unmatched_records`
* `error_count`

# 10. 当前实现限制

当前版本已经可以联调，但仍有一些明确限制：

1. 大模型理解层尚未正式接入，只完成了规则执行层；
2. `fetcher.py` 仍是模拟数据，尚未接真实业务系统 API；
3. 业务数据默认只支持一个数值字段 `value`；
4. 表头识别采用“前 10 行扫描”的简化策略；
5. 当前规则层虽然支持多级表头和换行列名，但复杂模板仍需进一步增强；
6. 暂未实现多目标列同时写入；
7. 暂未实现多工作表批量写入；
8. 暂未实现异步队列和长任务进度通知。

---

# 11. 后续建议

后续建议按以下顺序推进：

1. 接入大模型解析层，支持用户自然语言触发更新；
2. 将 `fetcher.py` 替换为真实业务系统 API 调用；
3. 支持“识别结果确认卡片”，由用户确认后再执行；
4. 支持业务数据返回结构配置化；
5. 增强 Excel 表头识别能力，兼容更多模板；
6. 支持多列更新；
7. 增加预览模式，先展示拟更新内容再执行写入；
8. 增加任务状态和进度查询能力；
9. 补充更多异常场景测试和真实 Excel 样本测试。
