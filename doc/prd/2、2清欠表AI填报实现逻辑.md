# Excel 固定任务更新服务 PRD

## 1. 实现目标

实现一个轻量 Excel 更新服务，供少量内部用户使用。

实现方式固定为：

* 用户上传一个 `.xlsx` 文件后，创建一个固定任务
* 任务详情页可多次发起“对比并更新”操作
* 每次操作基于该任务当前最新版本文件执行
* 每次操作完成后记录明细，并在有成功更新时生成一个新版本文件
* 原始文件、最新版本、历史版本均存储于 MinIO，可下载

本系统不是通用表格编辑器，只支持“按匹配列定位行，再更新目标列单元格”的固定模式。

---

## 2. 运行模型

### 2.1 固定任务

* 一个上传的 Excel 文件对应一个固定任务
* 任务创建后不再变化
* 后续变化体现在：操作记录、版本记录、当前最新版本指针

### 2.2 更新操作

一次更新操作由以下参数组成：

* `sheet_name`
* `match_column`
* `target_column`
* `updates[]`

其中 `updates[]` 的每一项至少包含：

* `match_value`
* `target_value`

### 2.3 版本规则

* 初始上传文件记为版本 `v1`
* 每次成功更新后生成新版本 `v2`、`v3`、`v4`...
* 后续操作始终基于当前最新版本执行，不基于原始文件重复执行

---

## 3. 必须满足的约束

1. 只更新目标列中的目标单元格，不修改 Excel 其他数据。
2. 同一任务下的更新操作必须串行执行。
3. 每次操作都必须记录逐条明细。
4. 每次成功更新后必须上传新版本文件到 MinIO。
5. 接口提交更新请求后立即返回，更新在后台异步执行。
6. 支持下载原始文件、当前最新版本、历史版本。
7. 若一次操作没有任何成功更新，可不生成新版本，但必须保留操作记录。

---

## 4. 更新规则

### 4.1 输入示例

* `match_column = A`
* `target_column = C`
* `sheet_name = Sheet1`

请求体中的 `updates[]` 示例：

```json
[
  { "match_value": "项目一", "target_value": "100" },
  { "match_value": "项目二", "target_value": "200" }
]
```

### 4.2 执行逻辑

1. 打开指定 `sheet`
2. 自动扫描 `match_column` 的有效数据区域
3. 构建 `match_value -> row_indexes[]` 映射
4. 遍历 `updates[]`

   * 未匹配到：记为 `not_found`
   * 匹配到多行：记为 `duplicate_match`
   * 唯一匹配：更新 `target_column + row_index`
5. 保存逐条结果
6. 若至少有 1 条更新成功，则保存新文件并生成新版本

### 4.3 更新边界

允许：

* 更新单元格值

禁止：

* 插行
* 删行
* 插列
* 删列
* 改 sheet 结构
* 改样式
* 改批注
* 改公式定义
* 全表重建

### 4.4 单元格更新规则

若匹配列为 `A`，目标列为 `C`，且 `项目一` 匹配到第 8 行，则只允许写入：

```text
C8 = target_value
```

其他单元格不得修改。

## 6. 数据模型（MySQL 建表）

约定：

* MySQL 8.0+
* 字符集：`utf8mb4`
* JSON 字段使用 `json`
* 主键使用 `bigint unsigned`
* 时间字段统一使用 `datetime(3)`
* 状态字段使用 `varchar`，避免 MySQL `enum` 后续变更困难

### 6.1 任务表 `excel_task`

```sql
CREATE TABLE `excel_task` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT,
  `task_name` varchar(255) NOT NULL COMMENT '任务名称',
  `original_file_name` varchar(255) NOT NULL COMMENT '原始文件名',
  `bucket_name` varchar(128) NOT NULL COMMENT 'MinIO bucket',
  `original_object_key` varchar(1024) NOT NULL COMMENT '原始文件对象路径',
  `latest_version_id` bigint unsigned DEFAULT NULL COMMENT '当前最新版本ID',
  `current_version_no` int unsigned NOT NULL DEFAULT 1 COMMENT '当前最新版本号',
  `status` varchar(32) NOT NULL DEFAULT 'active' COMMENT 'active/archived',
  `created_by` varchar(64) DEFAULT NULL COMMENT '创建人',
  `created_at` datetime(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  `updated_at` datetime(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  PRIMARY KEY (`id`),
  KEY `idx_status` (`status`),
  KEY `idx_created_at` (`created_at`),
  KEY `idx_latest_version_id` (`latest_version_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='Excel固定任务表';
```

### 6.2 版本表 `excel_task_version`

```sql
CREATE TABLE `excel_task_version` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT,
  `task_id` bigint unsigned NOT NULL COMMENT '所属任务ID',
  `version_no` int unsigned NOT NULL COMMENT '版本号，从1开始递增',

  `file_name` varchar(255) NOT NULL COMMENT '文件名',
  `file_size` bigint unsigned NOT NULL DEFAULT 0 COMMENT '文件大小，字节',
  `checksum_md5` varchar(64) DEFAULT NULL COMMENT '文件MD5',
  `source_operation_id` bigint unsigned DEFAULT NULL COMMENT '来源操作ID，原始版本可为空',
  `is_current` tinyint(1) NOT NULL DEFAULT 0 COMMENT '是否当前最新版本',
  `created_at` datetime(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),

  UNIQUE KEY `uk_task_version_no` (`task_id`, `version_no`),

```

### 6.5 建表补充约束

实现时补充以下规则：

1. `excel_task_version` 中同一 `task_id` 的 `version_no` 必须唯一。
2. `excel_task` 的 `latest_version_id` 应始终指向该任务当前 `is_current=1` 的版本。
3. 创建新版本时，需要先将旧版本 `is_current` 更新为 `0`，再将新版本标记为 `1`。
4. 同一任务下，同一时刻只允许一个 `excel_operation.status='running'`。该约束通过业务锁实现，不通过唯一索引实现。

## 9. API 设计

接口定义按 FastAPI 风格设计，返回体统一使用 JSON。除文件上传接口外，其他接口统一返回 `application/json`。

通用约定：

* 成功响应结构：`code + message + data`
* 失败响应结构：`code + message + error`
* 时间字段统一使用 ISO8601 字符串，示例：`2026-04-16T10:30:25.123+08:00`
* ID 字段统一使用整数
* 分页默认使用：`page`、`page_size`
* 下载接口返回下载地址，不直接在业务接口中回传文件二进制

### 9.0 通用响应结构

#### 成功响应

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

#### 失败响应

```json
{
  "code": 40001,
  "message": "invalid request",
  "error": {
    "field": "sheet_name",
    "reason": "sheet not found"
  }
}
```

#### 通用错误码

| code  | 含义     |
| ----- | ------ |
| 0     | 成功     |
| 40000 | 请求参数错误 |
| 40001 | 业务校验失败 |
| 40004 | 资源不存在  |
| 40900 | 状态冲突   |
| 50000 | 系统异常   |
|       |        |

  "message": "ok",

#### 失败场景

* 文件为空
* 文件扩展名不是 `.xlsx`
* MinIO 上传失败

### 9.3 发起一次更新操作

`POST /api/excel-tasks/{task_id}/operations`

说明：

* `task_id` 用于确定要更新哪个 Excel 文件任务
* `sheet_name` 用于确定更新该 Excel 文件中的哪个工作表（sheet）
* `sheet_name` 不是文件名，而是 Excel 底部标签页名称，例如：`Sheet1`、`3月产值`、`项目清单`

#### Path 参数

| 字段名     | 类型  | 必填 | 说明    |
| ------- | --- | -- | ----- |
| task_id | int | 是  | 任务 ID |

#### 请求体字段

| 字段名           | 类型     | 必填 | 说明               |
| ------------- | ------ | -- | ---------------- |
| sheet_name    | string | 是  | 工作表名称            |
| match_column  | string | 是  | 匹配列，如 `A`、`AA`   |
| target_column | string | 是  | 目标更新列，如 `C`、`AB` |
| updates       | array  | 是  | 待更新数据列表，不能为空     |
| submitted_by  | string | 否  | 提交人              |

#### `updates[]` 子项结构

| 字段名          | 类型                 | 必填 | 说明                      |
| ------------ | ------------------ | -- | ----------------------- |
| match_value  | string             | 是  | 匹配值，用于匹配 `match_column` |
| target_value | string/null/number | 否  | 写入目标单元格的值               |

#### 请求体示例

```json
{
  "sheet_name": "3月产值",
  "match_column": "A",
  "target_column": "C",
  "submitted_by": "zhangsan",
  "updates": [
    { "match_value": "项目一", "target_value": "100" },
    { "match_value": "项目二", "target_value": "200" }
  ]
}
```

#### 成功返回体

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "operation_id": 1001,
    "operation_no": "OP202604160001",
    "task_id": 1,
    "status": "pending",
    "submitted_at": "2026-04-16T11:20:00.000+08:00"
  }
}
```

#### 参数校验规则

* `sheet_name` 必须存在于当前最新版本的工作表列表中
* `match_column`、`target_column` 必须是合法 Excel 列名
* `updates` 不能为空
* `updates[].match_value` 不能为空字符串
* 单次 `updates` 数量建议限制，例如不超过 1000 条

## 10. 后端实现方式

实现目标是简单、稳定、可维护

推荐实现：

* FastAPI：接口服务
* MySQL：持久化任务、版本、操作、明细
* MinIO：文件存储
* openpyxl：Excel 更新引擎
* 独立 worker 进程：轮询 `pending` 操作并执行

### 10.1 worker 流程

1. 查询一条 `pending` 操作
2. 获取该 `task_id` 的执行锁
3. 将操作状态改为 `running`
4. 读取该任务当前最新版本
5. 从 MinIO 下载文件到本地临时目录
6. 使用 openpyxl 执行更新
7. 记录逐条结果
8. 若存在成功更新，则保存新版本文件并上传 MinIO
9. 创建版本记录，更新任务最新版本指针
10. 更新操作状态与统计信息
11. 释放锁

### 10.2 串行控制

同一任务下同时只能执行一个操作。

本期不接入 Redis，统一使用 MySQL 串行控制。

建议实现：

* worker 拉取 `pending` 操作后，先开启数据库事务
* 对 `excel_task` 对应任务行执行 `SELECT ... FOR UPDATE`
* 在同一事务内检查该任务是否已有 `running` 操作
* 若无，则将当前操作状态更新为 `running`
* 提交事务后开始实际处理 Excel
* 处理完成后再开启新事务写回操作结果、版本信息、任务最新版本指针

约束：

* 同一时刻，同一 `task_id` 只能有一个 worker 成功拿到任务行锁并完成状态切换
* 不使用应用内内存锁作为最终方案，避免多进程下失效
* 若 worker 异常退出，需依赖定时补偿任务扫描长时间处于 `running` 的操作并转为 `failed` 或重新入队

---

## 11. 关键实现要求

1. 每次执行必须基于任务当前最新版本，而不是原始文件。
2. 扫描范围由程序自动识别，不再由请求方传入起始行。
3. 只对命中的目标单元格写值，不允许重建工作簿。
4. `request_payload` 保存原始请求体，便于回放与排查。
5. `summary_payload` 保存汇总结果，减少前端重复计算。
6. 操作失败时保留 `error_message`。
7. 本地临时文件在执行结束后删除。
8. 如果操作没有任何成功更新，不创建新版本，但操作记录必须写入完成状态。

### 15.9 实际执行建议

* 一次只发一步提示词

* 本地跑通后再进入下一步
* 第五步核心 Excel 更新逻辑建议单独开一次会话处理

* 先做到第 4 步，拿到“可上传任务、可查询详情、可创建 pending 操作”的后端
* 再集中做第 5 步核心更新逻辑

这样能最大程度降低 Codex 在现有项目中大范围误改的风险。
